- **CRITICAL #1** `app/api/deps.py` – `must_change_password` se enforce-a u

  `/users/me/password` i `/auth/logout`.

  /saas-tenants/{id}` sad u jednoj transakciji briše sve dijete-tablice
  prije saas_tenants reda.
- **CRITICAL #3** `app/api/v1/endpoints/search.py` – escape `%`, `_`, `\\`

- **HIGH #4** `app/db/repositories/repos.py` – dodana fali-jedini `or_`
  import (krasao build prije).
- **HIGH #5** Nova **dashboard cache invalidation** infrastruktura:

  - `BaseRepository.invalidates_dashboard_cache = True` flag na 6 repoa
    (nekretnine, units, zakupnici, ugovori, maintenance, racuni)
  - svaki create/update/delete na tim tablicama briše cache za current
    tenant – frontend više ne čeka 30 s da vidi novu nekretninu/ugovor.
- **HIGH #6** `app/services/notification_service.py` – emaili idu samo
  članovima trenutnog tenanta (kroz `tenant_memberships`), ne više svim
  adminima u SaaS-u.

  trenutnom tenantu prije nego servira fajl.
- **HIGH #8** `app/api/v1/endpoints/{users,tenants}.py` – kod kreiranja
  korisnika/poziva temp lozinka se šalje email-om kad je SMTP
  konfiguriran; samo ako pošiljka ne uspije se vraća u response
  (`delivery: "email" | "response"`).
- **HIGH #9** `app/main.py` scheduler loop – koristi `CURRENT_TENANT_ID.set/reset`
  s tokenom umjesto resetiranja na None.
- **MEDIUM #10** `frontend/src/shared/api.js` – split na `apiClient`
  (10 s timeout) i `aiClient` (120 s). AI parse, PDF export, CSV
  import/export, agent rute koriste `aiClient`. Hung CRUD request više
  ne smrzava UI 2 min.
- **MEDIUM #11** `app/main.py activity_logger` – DB upis se okida kroz
  `asyncio.create_task` (fire-and-forget), s pending-set referencama da
  task ne padne GC. Latencija po requestu minus jedan SQL roundtrip.
- **MEDIUM #12** `app/db/repositories/base.py` – `find_all` ima hard cap

## 3. Server-side requirements (provjeri prije pull-a)

`backend/requirements.txt` je nadopunjen s 3 nove ovisnosti koje su prije
bile tranzitivne, ali brew Python 3.12 ih je trebao eksplicitno:

```
greenlet>=3.0.0     # SQLAlchemy async bridge
bcrypt>=3.2.0,<4.0  # passlib bcrypt backend (4.x je breaking)
aiosqlite>=0.20.0   # samo za dev SQLite; prod koristi asyncmy
```

`weasyprint>=62.3` je već bio tu. Native libovi za WeasyPrint su
dokumentirani u DEPLOY.md (`libpango-1.0-0 libpangoft2-1.0-0 libharfbuzz0b
libcairo2 libgdk-pixbuf-2.0-0`) — provjeri da su instalirani na server-u.

## 4. Deploy koraci (predloženi redoslijed)

1. **Backup** trenutne baze i `templates/` direktorija – noviji template-i
   imaju različitu klasu/strukturu.
2. **Pull / scp** kod na server.
3. **`pip install -r requirements.txt`** (povući će greenlet/bcrypt/aiosqlite).
4. **Alembic migracija**:
   ```bash
   cd /opt/riforma/backend
   source .venv/bin/activate
   alembic upgrade head
   ```
   Treba pokazati `Running upgrade 018_password_change_tracking →
   019_property_area_split`.
5. **Restart backend** (systemd / docker / pm2 — što god se koristi).
6. **Frontend build & deploy** – `npm run build` u `frontend/`, push
   `build/` na CDN/Apache.
7. **Smoke test** poslije deploy-a:
   ```bash
   curl -s https://<prod-domain>/health   # → {"status":"ok"}
   ```
   Login → dashboard → otvori nekretninu → kreiraj/uredi nekretninu s
   novim poljima → preuzmi portfolio + contracts PDF.

## 5. Rollback plan

Ako migracija pukne:
```bash
alembic downgrade 018_password_change_tracking
```
Polja `povrsina_objekta` i `povrsina_zemljista` su nullable bez default-a
i bez FK-a, pa downgrade ne dira postojeće podatke. Frontend će ignorirati
polja (svi referencirajući access-i su preko `?` ili truthy check).

Status pill class promjene u `contracts_report_pdf_service.py` i
`property_detail_pdf_service.py` su backward-inkompatibilne sa STARIM
templatima — ako vraćaš samo backend (a templati ostanu novi), bit će
neutralni pill-ovi za sve. Vrati i templates u istom rollbacku.
